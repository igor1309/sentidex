Спросим эксперта — gpt-5.4 🙂

makefile — это не устаревший формат, а вполне нормальный orchestration layer для локальной разработки.

Проблема обычно не в make, а в том, что именно в него запихнули.

Как я бы это оценил:

Когда make — хороший выбор
 • нужно дать проекту короткие стандартные команды: make dev, make test, make lint, make build
 • у тебя polyglot-репа: где-то Go, где-то Node, где-то Python, Docker, миграции, codegen
 • хочется один entrypoint для команды, независимо от внутренностей
 • проект запускается в Unix-like окружении и команда работает у всех одинаково

В этом смысле make очень хорош: он простой, везде есть, легко читается, и не привязывает тебя к одному стеку.

Когда начинаются претензии к make
Обычно люди имеют в виду не то, что make “старый”, а что он:
 • плохо подходит для сложной бизнес-логики в скриптах
 • капризен по синтаксису
 • не очень удобен для кроссплатформенности, особенно с Windows
 • исторически заточен под dependency-based build, а не под DX-оркестрацию
 • с ростом проекта Makefile превращается в помойку из shell-команд

И это правда. Если у тебя в Makefile уже:
 • много условий,
 • циклы,
 • парсинг env,
 • цветной вывод,
 • ветвления по ОС,
 • хитрая логика запуска сервисов,

то это уже сигнал, что make используется не по своей сильной стороне.

Нормальная современная позиция такая
 • make как тонкий фасад над командами — отлично
 • make как место для сложной логики — так себе

То есть вот так — хорошо:

dev:
 docker compose up

test:
 pytest

lint:
 npm run lint

А вот когда там 300 строк shell-магии — уже лучше вынести в:
 • bash-скрипты
 • just
 • task
 • npm scripts / pnpm
 • mage для Go
 • tox / nox для Python
 • turbo / nx для монорепы JS/TS

Мой практический вывод
Если у тебя:
 • backend / infra / docker-centric проект
 • команда сидит на macOS/Linux
 • нужен просто удобный entrypoint

то make dev — вообще ок, это не “legacy”.

Если у тебя:
 • сложная кроссплатформенная среда
 • heavy JS/TS monorepo
 • много логики вокруг тасков
 • нужны dependency graphs, caching, affected builds

тогда специализированный toolchain действительно лучше, чем make.

Хороший компромисс
Оставить внешний интерфейс через make, а сложность спрятать внутрь:

dev:
 ./scripts/dev.sh

Тогда:
 • у разработчика короткая команда make dev
 • логика живет в нормальном bash/python/node-скрипте
 • проект не зависит от вкусов каждого участника команды

Итог:
make не устарел. Он просто не должен быть твоим языком приложенческой логики. Как тонкий слой команд — очень удачный инструмент до сих пор.
