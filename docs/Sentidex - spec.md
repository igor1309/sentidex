# 📑 Спецификация решения: Telegram → Markdown Processing Queue

## Цель

Создать лёгкий личный воркфлоу для временного сохранения и подготовки важных сообщений из Telegram к последующей обработке, который обеспечивает простоту, скорость и доступность с разных устройств.

**Ключевая идея**: Это очередь обработки идей, предназначенная для достижения нуля, а не постоянное хранилище знаний.

---

## Основные принципы

- **Простота**: минимум действий от пользователя (только пересылка).
- **Скорость**: автоматическая обработка и сохранение без ожиданий.
- **Прозрачность**: все шаги отражаются в ежедневных и еженедельных сводках.
- **Достаточная приватность**: использование только Telegram и приватного GitHub-репозитория.
- **Минимум артефактов**: дубликаты не портят оригинал, а помечаются отдельными квитками.
- **Обработка до нуля**: система предназначена для временного хранения и последующего использования/удаления идей.

---

## Поток работы

1. **Пересылка**  
   Пользователь пересылает сообщение в выделенный канал/бот.

2. **Автоматическое обогащение**  
   Сообщение получает:  
   - краткое summary (1–2 предложения) для быстрой оценки при обработке,  
   - список тегов для группировки похожих идей,  
   - короткий, файлово-безопасный заголовок для имени файла,  
   - сохранение всех ссылок,  
   - отметку о наличии медиа,  
   - язык сохраняется как в оригинале.

3. **Создание файла**  
   - Формат: Markdown.  
   - Имя файла: краткий, файлово-безопасный заголовок (генерируется ИИ) и UUID для уникальности. Формат: ai-title-uuid.md.  
   - Временное хранилище: приватный GitHub-репозиторий.  
   - Содержимое:  
     - front matter: `id`, `created_at`, `source_url`, `summary`, `tags`, `has_media`, `links`, `language`.  
     - тело: полный текст сообщения.

4. **Обработка дубликатов**  
   - Проверка по `source_url`.  
   - Если совпадение найдено:  
     - оригинал не изменяется,  
     - создаётся **квиток-дубликат**.  
   - Имя квитка: `DUPL_summary + UUID.md`.  
   - Содержимое квитка:  
     - front matter: `id`, `created_at`, `source_url`, `original_ref`, `is_duplicate: true`.  
     - тело: «дубликат, см. оригинал».

5. **Сводки и напоминания**  
   - **Ежедневная** (вечером): список новых файлов и квитков за день для планирования обработки.  
   - **Еженедельная** (выходные): список всех необработанных файлов и квитков как напоминание о накопившейся очереди.  
   - Формат элемента в сводке:  
     - имя файла,  
     - summary,  
     - теги,  
     - ссылка на `source_url`.  
   - Дубликаты включены в сводки и помечены `DUPL`.

6. **Обработка и очистка**  
- Пользователь периодически просматривает очередь.  
- Обработанные/использованные идеи удаляются из репозитория.  
- Цель: регулярное достижение нулевой очереди.  

---

## Роли элементов

- **Оригинальный файл**: подготовленная к обработке идея.  
- **Квиток-дубликат**: отметка о повторной встрече источника.  
- **Сводки**: напоминания о необработанных идеях и планирование работы.  
- **GitHub-репозиторий**: временная синхронизированная очередь обработки.  

---

## Критерии достаточности

- Система работает в два шага: «переслал → получил сводку».  
- Файлы доступны через GitHub с любых устройств.  
- Дубликаты видны, но не засоряют структуру.  
- Пользователь не выполняет лишних действий, кроме пересылки.  
- **Очередь регулярно достигает нуля** через обработку и удаление использованных идей.  
- **Высокий процент использования** сохранённых идей (не накопление ради накопления).
